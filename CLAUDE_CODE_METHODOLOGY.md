# Как эффективно работать в Claude Code — методология проекта
### На основе официальных best-practices и курсов Anthropic. Это «как мы работаем»; `.claude/CLAUDE.md` — «по каким правилам».

---

## 0. Мышление: Claude — младший инженер, не генератор кода

Главный сдвиг, который рекомендует Anthropic: относиться к Claude Code как к **способному младшему инженеру с инструментами, памятью и итерацией**, а не как к «волшебной кнопке, которая выдаёт код». Из этого следует всё остальное: ему нужен контекст (как онбординг), чёткие правила (как код-стайл команды), возможность исследовать и проверять свою работу, и точки сохранения, чтобы не терять прогресс.

Claude Code намеренно **низкоуровневый и неопинионированный** — он не навязывает воркфлоу. Поэтому методология (этот файл) — наша ответственность, и она важнее, чем сам факт наличия инструмента.

---

## 1. Базовый цикл: Explore → Plan → Code → Commit

Канонический воркфлоу Anthropic. **Самый частый провал новичков — сразу просить код.** Правильно:

1. **Explore.** Сначала попроси Claude *исследовать* (прочитать релевантные файлы, доки через Context7, понять текущее состояние) — **без написания кода**. Можно явно: «не пиши код пока».
2. **Plan.** Попроси сформулировать стратегию/план. Используй **plan mode** (см. §2). На сложных задачах — на Opus.
3. **Code.** Только после согласованного плана — реализация. Sonnet.
4. **Commit.** Git-коммит с понятным сообщением + обновление progress-файла.

Это зеркалит то, как думают сеньоры: глобальные принципы + локальные ограничения, разведка перед стройкой.

---

## 2. Режимы: thinking vs planning

- **Plan mode** — Claude исследует и формирует план, **ничего не меняя**. Включай для любой нетривиальной задачи (новая система, рефактор, архитектурное решение). Сначала план → твоё «ок» → исполнение.
- **Extended thinking** («think», «think hard», «ultrathink») — для трудных рассуждений внутри шага (сложная логика стейт-машины, разбор гонок/багов). Чем сложнее — тем больше «бюджет мышления».

Правило: **архитектуру и план — на Opus в plan mode; реализацию — на Sonnet.**

---

## 3. Контекст — это инфраструктура

Claude автоматически тянет контекст в каждый промпт; это стоит времени и токенов. Управляй им:

- **`CLAUDE.md` авто-загружается** в начале сессии — это идеальное место для конвенций, команд, архитектуры, гард­рейлов. Держи его **тугим и актуальным** (раздутый CLAUDE.md ест контекст и теряется в шуме). Периодически проси Claude предложить, как его сократить/улучшить.
- **Прогрессивное раскрытие через Skills** — детальные инструкции (как развернуть, как тестировать) живут в `SKILL.md` и подгружаются только при вызове, а не висят в контексте постоянно.
- **Шумную работу — в сабагентов.** Чтение больших файлов, разведка кодовой базы, сравнение вариантов — делегируй сабагенту с отдельным контекстным окном, чтобы не засорять главный (см. §6).

---

## 4. ⭐ Harness для долгих сессий — система «не сломаться на пол пути»

Это ключевой раздел. Anthropic показал (пост «Effective harnesses for long-running agents»), что даже фронтир-модель в цикле через много контекстных окон проваливается двумя способами:

- **«One-shot всё сразу»** → выходит из контекста посреди реализации → следующая сессия наследует наполовину сделанную, недокументированную фичу и тратит время на разгадывание.
- **«Преждевременная победа»** → поздний агент видит, что прогресс есть, и объявляет проект готовым.

Решение Anthropic — двухчастный harness (**initializer-агент** + **coding-агент**) и набор артефактов. Мы адаптируем его 1-в-1:

### 4.1 Артефакты (создаёт initializer, см. `prompts/00-initializer.md`)
- **`harness/feature_list.json`** — полный список фич, каждая со `"passes": false`. **JSON, а не Markdown** — модель реже портит/переписывает JSON. Жёсткое правило: **агенты ТОЛЬКО меняют поле `passes`; удалять или редактировать фичи/тесты запрещено** (это ведёт к пропущенной/багованной функциональности).
- **`harness/claude-progress.md`** — журнал: что сделано, что дальше, известные проблемы.
- **`harness/init.sh`** — запускает сборку/смоук-тест (для нас: `rojo build` + прогон TestEZ через Lune/Open Cloud + базовый плейтест через Studio MCP).
- **Начальный git-коммит** — точка отсчёта.

### 4.2 Ритуал начала КАЖДОЙ сессии (coding-агент)
1. `pwd` — убедиться в рабочей директории.
2. Прочитать `claude-progress.md` и `git log --oneline -20` — войти в курс дела.
3. Прочитать `feature_list.json`, выбрать **одну** фичу с наивысшим приоритетом, у которой `passes:false`.
4. Запустить `init.sh` и сделать **базовый смоук-тест** (игра не сломана?) — ДО реализации новой фичи. Если что-то сломано — сначала чинить.
5. Реализовать **одну** фичу.
6. **Self-verify end-to-end** (см. §5) — пометить `passes:true` только после реальной проверки.
7. Git-коммит с описательным сообщением + запись в `claude-progress.md`.

### 4.3 Правила «чистого состояния»
В конце сессии код должен быть в состоянии «годно к мёрджу в main»: без серьёзных багов, опрятно, документировано — так, чтобы следующий агент мог начать новую фичу, не разгребая чужой беспорядок. Git-коммиты — это и точки восстановления (можно откатить плохое изменение).

> Открытый вопрос (по Anthropic): иногда лучше специализированные агенты (тест/QA/cleanup), чем один универсальный. У нас это покрыто сабагентами (§6) и `/review` (§7).

---

## 5. Тестирование: фича «passing» только после end-to-end

Anthropic особо отмечает: Claude склонен пометить фичу готовой без полноценной проверки — делает unit-тесты или `curl`, но не проверяет работу end-to-end как пользователь. Для веба они дают браузерную автоматизацию (Puppeteer MCP) и тестируют как человек.

**Наш аналог для Roblox:**
- Unit-логика → **TestEZ**, гоняется в реальном движке через **Open Cloud Luau Execution** в CI.
- End-to-end → **плейтест через Studio MCP**: агент реально запускает игру, проверяет «спавн → батарейка → генератор → пробуждение врага → выживание» как игрок.
- Никаких `passes:true` без прохождения end-to-end. Это прямое правило для всех агентов.

## 6. Сабагенты: изолированный контекст + структурный вывод

(Курс Anthropic «Introduction to Subagents».) Сабагент — это специализированный инстанс Claude **со своим контекстным окном**. Используй, чтобы:
- держать главный разговор чистым (разведка, ревью, чтение больших файлов уходят в сабагента);
- получать **структурированный вывод** (например, отчёт ревью безопасности по чек-листу) и **репорт препятствий** (агент сообщает, где застрял, а не молча выдумывает).

**Когда НЕ делегировать:** мелкая задача, где накладные расходы на сабагента больше пользы. Наши сабагенты — в `.claude/agents/` (initializer, security-reviewer, data-engineer, round-systems, performance-auditor, asset-pipeline).

**Agent Teams (окупает Max 20x):** несколько агентов параллельно под координацией Opus — один пилит систему, другой UI, третий данные, четвёртый ассеты. Каждый со своим контекстом.

## 7. Skills, команды, hooks — три разных инструмента

(Курс «Agent Skills» + «Claude Code in Action».)

- **Skill** (`SKILL.md` с frontmatter) — переиспользуемое поведение, которое Claude **сам подхватывает по контексту** или по `/имя`. Критично: **надёжное trigger-описание** (от него зависит срабатывание) и `allowed-tools` (ограничение инструментов), прогрессивное раскрытие (тело грузится только при вызове).
- **Slash-команда** (`.claude/commands/*.md`) — ты вызываешь **явно** (`/session-start`, `/ship`). Детерминированный воркфлоу под твоим контролем.
- **Hook** — код вокруг событий жизненного цикла. События: `PreToolUse`, `PostToolUse`, `Notification`, `Stop`, `SubagentStop`, `PreCompact`, `UserPromptSubmit`, `SessionStart`. У нас: авто-`StyLua`/`Selene` после правок (`PostToolUse`), напоминание про progress-файл на `Stop`/`PreCompact`, чтение `init.sh` на `SessionStart`.

Правило выбора: нужно явное управление временем запуска → команда; должно применяться автоматически по контексту → skill; должно срабатывать вокруг событий инструментов → hook.

---

## 8. Permissions = программируемая безопасность

Консервативная модель разрешений Claude Code — это не трение, а программируемая безопасность. Определи границы явно в `.claude/settings.json` (что можно без спроса, что требует подтверждения).
- `--dangerously-skip-permissions` — только в песочнице/контейнере без сети (для рутины вроде линта/бойлерплейта). Иначе риск (вкл. prompt injection).
- **MCP — это исполнение кода в твоей Studio.** Незнакомые серверы — ревью + read-only режим.
- Tooling формирует поведение: Claude рассуждает только о тех инструментах, которые понимает. Чем чище твой тулчейн (Rokit/Rojo/Selene/StyLua) — тем предсказуемее агент.

---

## 9. Мульти-модельность (опционально)

Claude Code умеет работать с другими моделями тремя способами: **Plugin** (CLI другой модели внутри Claude Code, напр. `/codex:review`), **MCP** (Claude Code зовёт другую модель как инструмент), **Router** (подмена эндпоинта). Полезно для второго мнения на ревью.

---

## 10. Анти-паттерны (чего НЕ делать)

- ❌ Просить код сразу, без Explore/Plan.
- ❌ «Сделай всю игру» / one-shot целой фазы.
- ❌ Объявлять готовым без end-to-end теста; помечать `passes:true` авансом.
- ❌ Удалять/править тесты или фичи в `feature_list.json`, чтобы «прошло».
- ❌ Раздувать `CLAUDE.md` (теряется сигнал).
- ❌ Длинная сессия без коммитов и записи прогресса (нечего восстанавливать).
- ❌ Доверять клиенту геймплейные решения (см. `docs/02`).
- ❌ Добавлять контент сверх текущей фазы (нарушение скоупа).

---

## 11. Чек-лист «не сломаться на пол пути»

- [ ] Есть `feature_list.json`, `claude-progress.md`, `init.sh`, начальный коммит.
- [ ] Каждая сессия начинается ритуалом §4.2 и заканчивается коммитом + записью прогресса.
- [ ] Одна фича за раз; чистое состояние в конце.
- [ ] `passes:true` только после end-to-end в Studio.
- [ ] CLAUDE.md тугой; правила вынесены в `docs/`.
- [ ] Plan mode (Opus) перед нетривиальной задачей.
- [ ] Сабагент-ревью (`/review`) перед мёрджем в main.
- [ ] Скоуп-гардрейлы соблюдены (только текущая фаза).
