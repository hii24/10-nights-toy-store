# 02 — Правила безопасности / анти-чит

## Аксиома
**Никогда не доверяй клиенту.** Считай каждый вызов RemoteEvent/RemoteFunction исходящим от враждебного клиента. Большинство заэксплойченных игр сделали одну ошибку: позволили LocalScript решать то, что должен решать сервер.

## Middleware на каждый RemoteEvent (в src/shared/Net, исполняется на сервере)
Порядок проверок ДО любой мутации состояния:
1. **Type-assert** каждого аргумента (строгая проверка типов).
2. **Whitelist** — значение входит в серверно-определённый допустимый набор.
3. **Bounds** — числовые входы ограничены разумными диапазонами.
4. **Rate-limit** — частота вызовов на игрока ограничена.
5. Только потом — игровая логика/мутация.

## Правила
- Вся валюта/прогресс/здоровье/«пойман ли»/позиционная авторитетность — собственность сервера.
- **Оптимистичный клиент, авторитетный сервер:** UI обновляется сразу для отзывчивости, но истина приходит от сервера.
- **Никогда `RemoteFunction:InvokeClient()` с сервера** (клиент может вечно подвесить серверный поток). Для server->client — `RemoteEvent`.
- Та же логика для `ProximityPrompt.Triggered` и `ClickDetector.MouseClick`.
- Repair/прогресс нельзя «доделать» мгновенно — сервер считает и rate-limit'ит.
- При провале валидации — **логировать и тихо игнорировать** вызов (агрессивный кик даёт ложные срабатывания). Кик — только за явно повторяющийся абьюз.
- Обфускация имён remote'ов в ReplicatedStorage НЕ защита (всё видно клиенту). Защита — серверная валидация.

## Экономика
Любой RemoteEvent, который пишет в DataStore без перепроверки на сервере, — это дыра (дюп/накрутка). Все начисления — серверно, после валидации.
