# 03 — Правила данных (ProfileStore) и монетизация

## Библиотека
**ProfileStore** (преемник ProfileService): авто-сейв + сессионные локи в одном ModuleScript; авто-сейв ~300с; разрешение конфликтов сессий через MessagingService; экспоненциальный backoff. Ставится через Wally.

## Железные правила (нарушение = потеря данных игроков)
- **При сбое загрузки профиля (сбой DataStore / rate-limit) — НЕ пускать игрока с дефолтными данными** (перезапишет реальный сейв). Показать ретрай-экран или кикнуть с понятным сообщением.
- Все DataStore-вызовы в `pcall`; ретраи с экспоненциальным backoff.
- Сейв на `Players.PlayerRemoving` **И** `game:BindToClose()`.
- Не чаще **1 сейва на ключ в 6 секунд**.
- Помнить лимиты DataStore: ~60 Get/мин + 6 на игрока; ~60 Set/мин + 6 на игрока.
- DataStore НЕ работает в локальном тесте Studio по умолчанию: включить API access (Game Settings -> Security) или mock-DataStore для тестов.
- При быстром реджоине профиль может грузиться до ~7с (сессионный лок) — обрабатывать, не паниковать.

## Что персистим (ProfileStore)
coins, toy tickets, XP/level, owned cosmetics, collectibles, badges, owned passes. (Сессионные апгрейды НЕ персистим.)

## Монетизация
- Passes + Developer Products через MarketplaceService.
- **`ProcessReceipt` ОБЯЗАН быть идемпотентным** — один receipt не выдаёт награду дважды. В `pcall` с ретраями. Выдача только серверно.
- Баланс: продаём статус/удобство/коллекцию (косметика, скины фонарика, toy pets, private servers, эмоции). НЕ продаём силу (не бессмертие, не «монстры не видят», не платный доступ к победе).
